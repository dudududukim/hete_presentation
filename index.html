문제 정의 · 측정 방법

  • 부분관측 격자맵에서 동적으로 출현하는 태스크를 발견·완료. 지표 = 완료율(완료/전체).
  • 로봇 6대(이종): 드론 2대(탐색 전용·수행 불가) · 지상 4대(태스크 실행, 이동비용 4–6배 높음).
  • 태스크는 관측되어야 할당·수행 가능 → 발견이 병목.
  • 제약: TIME_MAX · 태스크 총수 · 에너지를 사전에 모름 → 런타임 관측값만으로 동작.
측정 = 중간발표본과 최종본을 동일 bench_grid 하네스(제공 main.cpp 시뮬 루프 충실 재현)·동일 시드로 paired 비교. simulator·main 완전 동일, schedular만 교체 → 차이는 순수 알고리즘.

2. 추가 알고리즘 · 드론 ① — 연속 커버리지 스코어러

  • 문제: 중간발표 NBV는 도착지 정보이득 ÷ 거리만 봄. ① 먼 unknown 포켓을 거리페널티가 과하게 눌러 영영 안 감. ② 탐색(NBV)과 순찰(LRW)이 별도 로직이라 전환 경계에서 비효율.
  • 해결: pick_coverage_target — 단일 점수 유효 미관측도 ÷ 경로비용으로 탐색·순찰 통합.
    • 5×5 시야의 가중합: 미탐색 = 1.0, 오래 안 본 known = (안 본 시간)/현재, 점령·벽 = 0.
    • → 초기엔 미탐색 탐색, 후반엔 오래 안 본 곳 재관측으로 자연 전환 — 드론이 평생 탐색만 하다 죽지 않음.
  • regime 적응: 맵크기·관측 벽비율·커버리지로 상황 감지 → 거리페널티 지수(대형·개활 = cost^0.5로 먼 포켓까지 도달) · 다양성 floor 자동. 맵크기 하드코딩이 아닌 런타임 신호 → 임의 맵에 일반화.
발견이 병목인데 중간발표 NBV는 먼 영역을 구조적으로 못 감 → 거리페널티를 상황별로 완화.

2. 추가 알고리즘 · 드론 ② — reserve-drone 시간 직렬화

  • 문제: 중간발표는 두 드론이 동시 탐색 → 둘 다 비슷한 시기에 에너지 ~100% 소진·사망. 그런데 태스크 절반은 후반에 출현(late-spawn) → 그땐 드론이 이미 죽어 센서 0 → 후기 태스크 영영 미발견.
  • 해결: 드론 2대 중 1대를 late_start까지 0에너지로 보류(hold) → 나머지 1대가 전반 탐색. late_start에 release → 후반 커버. 두 드론을 시간축으로 직렬화해 센서 생존 기간을 늘림.
  • 타이밍 유도: late_start = TIME_MAX × 3/16. TIME_MAX은 인자로 안 주어짐 → 관측한 드론 초기 에너지 ÷ 6로 역산(드론 init 에너지 = TIME_MAX×6). 오라클 없음, time 배율이 바뀌어도 자동 보정.
  • 왜 3/16: 너무 일찍 풀면 두 드론이 같은 시기에 죽어 후반 공백, 너무 늦으면 reserve가 노는 시간 과다. 측정상 3/16이 최적.
핵심은 "언제 센서를 켜 두나"(타이밍) — 후기 spawn 태스크를 볼 눈을 후반까지 남김.

2. 추가 알고리즘 · 워커 ① — 유휴 워커의 보조 발견 (드론 전멸 시 가속)

  • 문제: 워커는 이동비용 4–6배라 탐색 ROI 음수 → 중간발표는 유휴 워커 정지(HOLD)(맞는 선택). 하지만 갈 수 있는 태스크가 없어 오래 멈춰 있는 워커는 그냥 노는 무한에너지 센서 낭비(워커는 거의 안 죽고, 시뮬은 시간이 다 차야 끝남).
  • 해결: 일정 시간 이상 멈춰 있고 인근이 오래(≥250 tick) 관측 안 됐으면 가까운(≤3셀) 미관측 셀로 살짝 보조 이동 → 이동 중 5–9셀 부수 관측으로 뒤늦게 생긴 태스크 발굴. 한 tick에 1명만, 에너지 30% 남겨 완료 임무 보호.
  • 드론 전멸 시 가속: 드론이 살아 있을 땐 멈춤 임계 65 tick(완료 임무 거의 방해 0). 드론이 모두 소진되면 워커가 유일한 센서라 30 tick으로 발견을 앞당김. 맵 크기로 나누지 않고 실행 중 상태로만 판단 → 일반화.
탐색이 손해인 워커를 오래 멈춰 있을 때만 발견에 투입 → 일반화되는 첫 양수 워커 레버(전 맵 회귀 0).

2. 추가 알고리즘 · 워커 ② — 미탐색 직행 · 선점 · 게이트

  • 미탐색 경계(frontier) 직행: 보조 이동 시 미탐색이 절반 넘게 남았으면, 이미 본 영역 대신 예산(8셀) 안의 미탐색 경계로 곧장 → 왜: 드론이 에너지로 못 닿은 새 영역을 워커가 직접 발굴.
  • 중간 재배치 선점 (10% 이상 더 싸야 교체): 이미 출발한 워커의 태스크를, 놀고 있는 더 가까운 워커가 10% 이상 싸게 끝낼 수 있으면 가져오고 기존 워커는 풀어 줌 → 왜: 이동 중 워커의 태스크 독점 해소. 10% 여유를 둬 오락가락·좁은 길 충돌 억제.
  • 에너지 실행가능 검사(잔여 97% 기준): 못 끝낼 태스크에 워커가 묶이는 것 방지.
  • 왕복 순회(lawnmower) 보강: 예비 드론 투입 뒤 빠진 영역을 고르게 덮음.
워커 추가의 공통 원리: 유휴 무한에너지 센서를 발견에 투입하되, 모든 발동을 런타임 게이트로 묶어 완료 임무를 해치지 않게.

4. 시도했던 방법들 ① — 채택 (측정으로 검증)

4. 시도했던 방법들 ② — 교훈적 기각

  • 드론을 일부러 흩어놓기 (서로 밀어내기 · 보로노이 영역분할 · 시작 시 모서리로 고정)
    • "두 드론이 같은 영역만 판다"는 시각적 직관.
      측정: 일반화 실패(모서리 고정 시 40×40 −6.77pp).
      교훈: 드론은 먼 영역을 방치가 아니라 늦게 덮음 — 경로 모양은 완료율과 무관, 병목은 협응이 아니라 에너지로 닿는 범위.
  • 먼 영역에 드론 강제 배정
    • 실행 중 "못 닿는 영역"을 감지해 드론을 먼 곳으로 강제 배정.
      4번 모두 효과 없음.
      교훈: 가까운 곳부터 훑는 게 물리적 최적 — 희소한 드론 에너지를 먼 저밀도에 쓰면 발견 손해.
  • 고정 임계값을 실행 중 신호로 대체
    • 고정 임계값 65를 "일반적"인 실행 중 신호로 대체.
      −2.71pp.
      교훈: 드물게 고정상수가 원리적 신호보다 더 잘 일반화.
새롭다고 다 개선은 아니다. 눈에 띄는 비효율도 막상 재보면 완료율과 무관한 경우가 대부분.

5. 한계 — 더 못 올리는 이유

  • 드론 2대로는 큰 맵을 다 못 본다
    드론 시야가 5×5로 좁아서, 맵이 커지면 정해진 시간 안에 전체를 훑지 못한다. 맵이 넓어질수록 못 본 영역이 늘어 완료율이 떨어진다 — 알고리즘 문제가 아니라 드론 수·시야의 물리적 한계.
  • 벽이 너무 많으면 갇힌 공간이 생긴다
    벽이 빽빽하면 맵이 여러 조각으로 끊겨, 아무리 잘 움직여도 들어갈 수 없는 방이 남는다. (벽 40%: 태스크를 7.6개 찾아도 길이 막혀 4.6개만 완료.)
  • 더 짜낼 수 있는 건 "무엇을 하나"가 아니라 "언제 하나"였다
    점수 매기는 법·비용·일 나누기 등 약 50가지를 바꿔봤지만 대부분 제자리거나 오히려 나빠졌다. 드론이 어디를 볼지 고르는 방식은 이미 거의 최선이었다.
  • 좋아 보이는 결과를 그대로 믿지 않았다
    운 좋은 몇몇 맵에서만 좋아 보이는 경우가 많아서, 새로운 맵 2000개 이상에서 한 곳도 나빠지지 않을 때만 진짜 개선으로 인정했다.
큰 맵·벽 많은 맵의 낮은 완료율은 실력 부족이 아니라 드론 수·시야·벽이라는 물리적 한계 때문. 더 넘으려면 드론끼리 영역을 나눠 협력하는 식으로 구조 자체를 새로 짜야 한다.
